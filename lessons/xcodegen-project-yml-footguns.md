## Footgun 1 — `minVersion` standalone for SPM packages does not exist

```yaml
# ❌ FAILS at xcodegen generate:
packages:
  Supabase:
    url: https://github.com/supabase/supabase-swift
    minVersion: "2.20.0"

# Output:
#   Parsing project spec failed:
#   Unknown package requirement: ["minVersion": "2.20.0", "url": ...]
```

`minVersion` is only valid in combination with `maxVersion` as a Range-Spec. Standalone-minimum is **`from`** (or its alias `majorVersion`):

```yaml
# ✅ CORRECT:
packages:
  Supabase:
    url: https://github.com/supabase/supabase-swift
    from: 2.20.0
```

Allowed standalone keys are `from`, `majorVersion`, `exactVersion` (alias `version`), `branch`, `revision`. Range-pairs are `minVersion + maxVersion`.

## Footgun 2 — Info.plist is regenerated on every `xcodegen generate`

If you write Custom-Keys directly into `Sources/<App>/Resources/Info.plist`, they get wiped on the next `xcodegen generate`. XcodeGen rewrites the file from `info.properties` in `project.yml` plus its standard defaults — anything not in `info.properties` is silently dropped.

```yaml
# project.yml — Custom-Keys MUST go here:
targets:
  Taxray:
    info:
      path: Sources/Taxray/Resources/Info.plist
      properties:
        CFBundleDisplayName: taxray
        # Custom runtime values populated via xcconfig variable substitution:
        SupabaseURL: "$(SUPABASE_URL)"
        SupabaseAnonKey: "$(SUPABASE_ANON_KEY)"
        # URL-Scheme registration so taxray://... routes back to the app:
        CFBundleURLTypes:
          - CFBundleTypeRole: Editor
            CFBundleURLName: app.taxray.ios.auth-callback
            CFBundleURLSchemes: [taxray]
```

Convention — never hand-edit the generated `Info.plist`. All changes go through `project.yml`. Optionally `.gitignore` the Info.plist (it is fully derivable), though committing it helps code-review.

## Footgun 3 — xcconfig treats `//` as comment

```
# ❌ Silently truncates the URL during build:
SUPABASE_URL = https://your-project.supabase.co
# At build time: SUPABASE_URL = "https:"  (rest is comment)

# ✅ Workaround — `$()` is an empty variable substitution that disappears:
SUPABASE_URL = https:/$()/your-project.supabase.co
# At build time: SUPABASE_URL = "https://your-project.supabase.co"
```

The `$()` trick is an established Xcode-community workaround. XcodeGen itself is not involved — this is plain Xcode-build-system behavior. Other URLs with `//` need the same treatment.

## Workflow tip — CLI build verification without `sudo xcode-select`

If you're debugging an iOS issue and can't easily run Xcode, the natural reflex is to guess from source. Stop and run a real build instead:

```bash
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  xcodebuild -project Taxray.xcodeproj -scheme Taxray \
  -destination 'generic/platform=iOS Simulator' -configuration Debug \
  CODE_SIGNING_ALLOWED=NO build 2>&1 | grep -E "error:|warning:|BUILD" | head -20
```

The `DEVELOPER_DIR` env-var replaces persistent `sudo xcode-select` configuration (useful when Command Line Tools is set as the active developer dir). `CODE_SIGNING_ALLOWED=NO` skips provisioning-profile setup, which is irrelevant for build verification. The `grep` filters out the verbose Swift driver output and shows just errors, warnings, and the final BUILD line.

This 30-60 second loop beats three rounds of "let me see the source code again" by an order of magnitude. Run it before asking anyone — including yourself — to inspect the source for hypothetical compile errors.

## When this does NOT apply

- **You don't use XcodeGen:** if your `.xcodeproj` is hand-maintained or generated by Tuist, Footguns 1 and 2 don't apply. Footgun 3 (xcconfig `//` comment) still applies — that's Xcode-build-system, not XcodeGen.
- **You don't use xcconfig files:** Footgun 3 only matters if you have URLs in xcconfig. Build-Settings in `project.yml` directly are fine.
- **Older Xcode without Command Line Tools collision:** if you don't have CLT installed alongside Xcode, `xcodebuild` works without the `DEVELOPER_DIR` override. The workflow tip is for the common-on-Mac case of both being present.

Related conventions — `lsn_swift_6_main_actor_singleton_deinit` (Swift-6 concurrency errors that surface on the first verified build) and `lsn_ios_supabase_magic_link_deep_link` (uses CFBundleURLTypes in info.properties as illustrated above).

## Anti-patterns

- **Hand-editing the generated `.pbxproj`:** XcodeGen overwrites it on every `xcodegen generate`. Edit `project.yml` instead. `.gitignore` the `.pbxproj` if you want zero ambiguity about source of truth.
- **Committing `.xcodeproj` to git:** merge conflicts in the binary-ish XML are painful. Generate it on demand via `xcodegen generate`. The `project.yml` plus source files are the durable artifact.
- **Guessing iOS build issues from shared source code without running xcodebuild:** three iterations of "this might be the issue, please re-test in Xcode" is a smell. Run `xcodebuild` once, see the actual error, fix once.
